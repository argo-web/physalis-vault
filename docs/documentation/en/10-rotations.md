# Secret rotation

**Rotation** periodically renews a credential. The key principle: you **change
the credential at the source** (on the target system), **then** update the value
in Physalis — it is not a simple `upsert` in the vault.

Two families:

- **Automatic** — an executor changes the credential at the source then reports
  the new value (databases, internal secrets, application hooks…).
- **Assisted reminder** — Physalis cannot change the credential for you (human
  password, third-party account): it **notifies** you, and you **generate /
  enter** the new value, which is then stored and versioned.

Rotation applies to several **places where secrets live**:

| Object | Where | Possible strategies |
|--------|-------|---------------------|
| **Environment secret** | a project's *Secrets* tab | Database, JWT, API key, Webhook, Reminder |
| **Service** (Stripe, OVH…) | *Access* tab | Assisted reminder (on its own credentials) |
| **Application account** | *Access* tab | Assisted reminder, **Webhook** (linked backend's hook) or **Database** (role/user of a managed database such as Supabase) |
| **Team / org vault entry** | *Vault* tab | Assisted reminder |

## Which strategy for which case?

| You want to rotate… | Strategy | How |
|---------------------|----------|-----|
| a **database** password (PG/MySQL role) | **Database** | automatic, self-rotation |
| an internal **JWT / session / encryption key** | **JWT Secret** | automatic, generated by Physalis |
| a Physalis **API Gateway key** | **API key** | automatic |
| an **account password** (admin/user) hashed by the app | **Webhook** | the app applies it via a hook |
| an **account of a managed database** (Postgres role or Supabase Auth user) | **Database** | Physalis connects (linked service admin) and applies it |
| a **third-party key / token** (Stripe, Mailgun…) no hook covers | **Reminder** | you change it at the source, then store it |

## Prerequisites

Rotation is **opt-in at the organisation level** and reserved for **paid plans**.
An org **ADMIN** or **OWNER** enables it in *Organisation settings → Advanced*.
While disabled, no rotation button appears and the cron ignores the organisation.

It also pauses when a **project is paused**.

## The "Rotation" button

Everywhere (secret, service, account, vault entry), a **single "Rotation"
button** opens a modal that combines:

1. **Configuration**: enable + **interval** (in days) + the **strategy** (for
   environment secrets).
2. **Immediate rotation**: for a reminder, a *generate / enter* section for the
   new value; for an automatic strategy, a **"Force"** button.

> The "Rotation" button only appears on items that look like a credential (the
> name contains `password`, `secret`, `token`, `key`, `jwt`…). A `PORT`, a public
> URL or a flag don't get it. All strategies stay selectable in the modal, with a
> **smart default** inferred from the name (a `*_PASSWORD` → Database, a
> `JWT_SECRET` → JWT, everything else → Reminder).

## Strategies (environment secrets)

### Database

**Self-rotation** of a PostgreSQL / MySQL role password: Physalis connects **as**
the user being rotated (with its current password, read from the injected `.env`)
and runs `ALTER … PASSWORD` — **no admin credential** is stored or used. Two
execution modes:

- **Agent on the VPS** *(default)* — for a **database internal to the project's
  Docker network**: the **agent** sidecar (the same as for backups) performs the
  rotation **locally**, then reports the new value to Physalis. **No external
  call** is needed.
- **Direct** — for a **managed database reachable over TCP+SSL** (Supabase, RDS,
  Neon…): Physalis connects **directly** (with the current password from the
  vault), runs the `ALTER`, verifies reconnection, then commits. **No external
  dependency** (no N8n). PostgreSQL for now.

| Field | Description |
|-------|-------------|
| `dbType` | `POSTGRESQL` or `MYSQL` |
| `dbHost` | host (Docker service name in Agent mode, public host in Direct) |
| `dbPort` | port (`5432`, `3306`…) |
| `dbName` | database name |
| `dbUser` | user whose password is rotated |

After the source change is confirmed, Physalis writes the new value (snapshotting
the old one into versioning) and triggers a **redeployment** so the application
reloads its `.env`.

### JWT Secret

Physalis **generates** a new random value itself (64 bytes), encrypts it, archives
the old one, then triggers a redeployment — **with no external intervention**.
Ideal for `JWT_SECRET`, `NEXTAUTH_SECRET`, `SESSION_SECRET`, internal encryption
keys…

### API Gateway key

Generates a new key in the project's **API Gateway**, updates the secret,
**immediately revokes** the old one, then redeploys. The secret must be linked to
an existing key (select API + key). Only for keys **issued by the Physalis
gateway** (not a third-party key like Stripe).

### Webhook (application-side hook)

For credentials that **only the application can apply** — typically an **account
password** hashed in the database by the app (admin, user). See the
**Rotation via hook (Webhook)** section below.

### Reminder (assisted)

Physalis **changes nothing at the source**. When due, it **notifies** the org
ADMIN / OWNER and shows a badge. You change the credential at the provider, then,
via the modal's **immediate rotation**, you **generate or enter** the new value:
Physalis stores it and archives the old one. Suited to third-party keys, tokens,
shared passwords.

## Rotation via hook (Webhook)

Rotating **admin/user accounts** is blocked by **hashing**: only the application
can correctly hash the password (bcrypt, argon2, salt, pepper…). Reproducing that
hashing in Physalis would be fragile and risk a lockout. The solution: **a hook
exposed by the application** that applies the new password with its own code.

**Principle**: Physalis (or the agent) **generates** a strong password and sends
it to the hook; the application **applies** it (hashes + updates its source) and
responds **2xx**; Physalis then **commits** the value it generated.

### The hook contract

The application must expose an endpoint that responds `2xx` once the credential is
applied:

```http
POST <hook-url>
Authorization: Bearer <token>        # optional but recommended
Content-Type: application/json

# for an environment secret:
{ "secretKey": "ADMIN_PASSWORD", "newValue": "<generated by Physalis>" }

# for an application account:
{ "user": "admin@example.com", "newValue": "<generated by Physalis>" }
```

- **`Bearer <token>`**: a shared secret. You set it in Physalis, and your hook
  verifies it. It's often a **token provided by the backend** (e.g. a Directus
  access token) that you paste; a *Generate* button is available if you prefer a
  dedicated secret.
- **`2xx` response** = applied → Physalis stores the value. Any other code =
  failure → nothing is committed (no drift).

### Modes (hook reachability)

- **Agent** — the hook is **internal** to the project's Docker network
  (e.g. `http://app:3000/internal/rotate`): the **agent** calls it. Common case
  of a non-exposed self-hosted client app.
- **Direct** — the hook is **reachable from Physalis** (public URL, automation
  platform): Physalis calls it directly.

### Where is the hook configured?

- **Environment secret** in Webhook strategy: the URL/token/mode are set **on the
  secret**.
- **Application account**: the hook is set **on the linked backend service** (see
  the **Application accounts** section). This way, several accounts on the same
  backend share a single hook.

### Example: Directus

Directus has no endpoint in this format; create a **Flow**:

1. *Settings → Flows → Create Flow*. Trigger **Webhook (POST)**, **Response Body =
   "Data of last operation"** → the URL `…/flows/trigger/<id>` is your hook URL.
2. *(auth)* a **Condition** operation: `{{$trigger.headers.authorization}}` equals
   `Bearer <your-token>`.
3. **Read Data** on `directus_users`, filter `email == {{$trigger.body.user}}` →
   get the `id`.
4. **Update Data** on `directus_users`, key = that `id`, payload
   `{ "password": "{{$trigger.body.newValue}}" }` (Directus hashes with argon2).

## Application accounts

An **Account** (*Access* tab) holds login credentials for the project's app. You
can **link** it to an **environment** (frontend) or a **service** (backend): its
URL follows from the link (single, synchronised source), which lets the browser
extension offer it on the right page.

For rotation, an account has three possible strategies:

- **Reminder** (assisted) — the default.
- **Webhook** — it **must be linked to a backend service whose hook is
  configured** (the hook lives on the service). "Force" runs the hook (Direct
  mode) or delegates it to the agent (Agent mode).
- **Database** — for an account of a **managed database** (Supabase…): it must be
  linked to a **"database" service** (see below). Physalis connects with the
  service's **DB admin** and applies the new password. Two **target types**:
  - **Database role** (PostgreSQL) → `ALTER ROLE`; the account identifier is the
    role name.
  - **Supabase Auth user** (`auth.users`) → `UPDATE auth.users` (bcrypt hash);
    the account identifier is the **email**.

## Services

A **Service** (*Access* tab) has three uses:

- **Third-party service** (Stripe, OVH…): a username + password. Its rotation is
  an **assisted reminder** on its own credentials.
- **Backend service**: often **just a URL** (username/password **optional**),
  carrying the **account rotation hook** for its linked accounts. The "Account
  rotation hook" section of the editor sets the URL, token and mode
  (Agent / Direct).
- **Managed database service** (Supabase, RDS…): check **"Database target"** and
  fill in `type / host / port / database` **plus a dedicated DB username and
  password** (the admin, e.g. `postgres` or `postgres.<ref>` via the Supabase
  pooler). ⚠️ These DB credentials are **distinct** from the credentials "at the
  top" (which document the **dashboard login**). Linked accounts using the
  **Database** strategy rotate through this admin connection.

## Step by step: rotate an account password via a hook

Typical case: an **admin** account of a client app, whose password is hashed in
the database by the application.

1. **Expose a hook on the app side**: an endpoint that receives `{ user, newValue }`,
   applies the new password (hashes it + updates the row) and responds `2xx`. (With
   Directus, a *Flow* — see the example above.)
2. **Create/edit the backend service** (*Access* tab): set its **URL**, enable
   **"Account rotation hook"** and enter the **hook URL**, **token** and **mode**
   (Agent if the hook is internal to the Docker network, Direct if reachable from
   Physalis). The service's credentials are optional.
3. **Link the account to the service**: in the account editor, *Linked to →
   Service →* your backend.
4. **Enable rotation on the account**: **Rotation** button → enable, interval,
   strategy **Webhook**. An indicator confirms the linked service has a hook.
5. **Test**: **"Force rotation"** button. In Direct mode, Physalis calls the hook
   immediately; if the hook responds `2xx`, the new value is stored and versioned.
   In Agent mode, the agent runs it on its next cycle.

> First verify the mechanism by pointing the hook at a test endpoint that returns
> `200` (e.g. webhook.site): you confirm the *generate → POST → store* cycle with
> no lockout risk, then plug in the real hook.

## Team and organisation vaults

Vault entries rotate via **assisted reminder** (generate/enter + the **3 latest**
values kept for revert). **Project** vault entries also appear in the org's
Rotation tab; **org** vaults are managed from the Vault tab.

## Immediate rotation & "Force"

From an item's modal, or from the organisation's **Rotation** tab:

- **Reminder / assisted** → *Immediate rotation* section: generate or enter the
  new value. A **blocking confirmation** reminds you that Physalis stores the value
  **but does not apply it at the source** — change it at the provider first.
- **Automatic strategy** (Database, JWT, API key, Webhook) → **"Force"** button:
  triggers the rotation now, outside the schedule.

## Organisation "Rotation" tab

A per-project overview of all active rotations (secrets, email keys, services,
accounts, project vaults). Each row has a **Rotation** button (config + immediate).
A project **OWNER** can **pause** all its rotations (useful during maintenance):

```http
PATCH /api/projects/<slug>/rotation/pause
{ "paused": true }
```

## Scheduling

Automatic rotations run at a configurable **off-peak hour** (default **2 AM UTC**):
the brief redeployment at the end of a rotation thus falls outside peak hours. The
**"Force"** button ignores this window.

## Status & monitoring

| Field | Description |
|-------|-------------|
| `rotationLastStatus` | `success`, `error` or empty |
| `rotationLastAt` | date of the last rotation |
| `rotationNextAt` | next scheduled run |

An email notification is sent to the **ADMIN / OWNER** on the **first failure** (no
spam afterwards). Every rotation is recorded in the audit log.

## History & revert

- **Environment secrets**: the old value is archived in the secret's full
  **versioning** (see *Secrets & categories*).
- **Services, accounts, vault entries** (no versioning): the **3 latest values**
  are kept so you can roll back.

## Troubleshooting

| Symptom | Likely cause / fix |
|---------|--------------------|
| **"Force" returns `Hook failed: …`** | The hook responded non-2xx or is unreachable. The message includes the code and the start of the response. Check the URL, the token (`Bearer`), and that the hook returns `2xx`. |
| **502 / the page hangs when forcing** | The hook doesn't respond in time. In **Direct** mode, the URL must be reachable **from Physalis** (not just from your machine). Check your Flow returns a response (Response Body configured). |
| **"Link the account to a backend service with a hook"** | The account is Webhook but its linked service has no hook configured. Set the hook **on the service** (Access tab). |
| **The account doesn't appear in the extension** | The extension offers a credential when the page URL matches. An account appears on the URL of its **linked target** (environment or service) — not on an unlinked URL. |
| **The "Rotation" button doesn't appear on a secret** | The name isn't recognised as a credential (`PORT`, URL, flag…). This is intentional. |
| **No automatic rotation triggers** | The cron runs off-peak (default 2 AM UTC). Use **"Force"** to test on demand. Also check the feature is enabled on the org and the project isn't paused. |

## Security

- **Self-rotation without an admin credential** (Database/Agent): the agent
  changes the password **of the account it uses**, never a superuser.
- **Hashing stays in the application** (Webhook): Physalis never reproduces an
  app's hash scheme.
- **Atomicity**: the new value is written only **after** the source change is
  confirmed → no drift between the source and the vault.
- **The cron never decrypts** a credential: it marks it "due" and delegates to the
  executor (agent, hook, or reminder).
